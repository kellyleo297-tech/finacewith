## 1. 产品定位与策略

### Q1：市面上有那么多记账产品，为什么还要做这个？你的差异化是什么？

**回答：**

传统记账产品（随手记、鲨鱼记账等）解决的是"记录"问题，但用户真正的痛点是三个：

1. **不知道钱去哪了** — 记了账但看不懂
2. **不知道还能花多少** — 没有实时预算反馈
3. **不知道怎么改善** — 没有个性化建议

MoneyMate 的差异化在于：**不是记账工具，是财务理解工具**。

| 传统记账 | MoneyMate AI |
|---------|-------------|
| 用户负责记 + 看 | AI 负责记 + 分析 + 建议 |
| 打开看数字 | 打开看到一句话：「你今天还能花 85 元」 |
| 月结时才焦虑 | 预算到 70% 就提醒 |
| 需要财务知识才能看懂报表 | AI 用自然语言解释你的财务状态 |

核心洞察：年轻人不缺记账的工具，缺的是一个能帮他们理解自己财务状况的"人"。AI 填补了这个角色。

---

### Q2：为什么定义目标用户是"学生和年轻白领"？这个群体的核心需求是什么？

**回答：**

选择这个群体有三个原因：

**1. 需求真实且高频**
- 收入固定但消费场景碎片化（外卖、奶茶、打车、订阅）
- 每月都在"月底没钱了"的循环里
- 有储蓄意愿但缺乏执行工具

**2. 决策链路短**
- 个人使用场景，不需要家庭协商、企业审批
- C 端产品，自传播性强

**3. 市场空白清晰**
- 高端：私人银行理财顾问（太贵）
- 低端：纯记账工具（太笨）
- 中间：缺少一个"AI 轻理财"的产品

核心需求不是"记账"，而是**回答一个每天的问题：我今天还能花多少钱？**

---

### Q3：MVP 阶段你选择了 5 个页面，怎么判断这些是 MVP 必须的？

**回答：**

判断标准是：**能不能跑通一个完整的用户价值闭环**。

我设计的闭环是：

```
设置月收入 → 记录支出 → 预算使用率计算 → 超支提醒 → AI 分析 → AI 建议
```

5 个页面分别对应这个闭环的节点：

| 页面 | 闭环角色 |
|------|---------|
| 引导页 | 输入初始条件（收入、储蓄目标） |
| 首页看板 | 回答"今天还能花多少" |
| 记账页 | 持续输入数据 |
| 统计页 | 可视化消费结构 |
| AI 助手 | 解释 + 建议 |

每个被砍掉的 V1 功能（语音记账、账单导入、多账户），我都会问自己：**没有它，闭环还能转吗？** 答案是能，那就放到 V1。

## 2. AI Agent 架构设计

### Q4：你为什么选择多 Agent 架构，而不是一个 Agent 什么都做？

**回答：**

本质上是一个**单一职责 vs 全能型**的权衡。我选择多 Agent 有三个理由：

**1. 可控性**
每个 Agent 的 System Prompt 只有 ~200 字，非常聚焦。如果用一个超级 Prompt，当一个场景出问题时，你很难定位是 prompt 的哪个部分导致的。

**2. 可评测性**
7 个 Agent 可以独立评测：
- 记账 Agent：金额准确率、分类准确率
- 风控 Agent：拦截率、误拦截率
- Supervisor：意图识别准确率

独立评测才能独立优化。一个全能 Agent 你没法知道它哪里差。

**3. 安全性**
理财场景有合规要求。把风控放在最后一道关卡，比其他 Agent 的代码耦合度为零。即使理财 Agent 出了问题，风控还能拦一道。

**但这个架构也有代价**：
- Supervisor 路由错误会导致后续全错（所以我加了规则层兜底）
- 多个 Agent 串联增加了延迟（所以 Supervisor 用规则匹配，不走 LLM 的 case 能做到 0ms）

---

### Q5：你的 7 个 Agent 是怎么划分的？依据是什么？

**回答：**

划分依据是**用户意图类型**，而不是技术上的功能模块。这是有意为之的——Agent 架构应该映射用户的 mental model，而不是开发者的 mental model。

```
用户心里想的            →  调用的 Agent
───────────────────────────────────
"帮我记一下"            →  记账 Agent
"我哪里花多了"           →  消费分析 Agent
"下个月怎么规划"         →  预算规划 Agent
"我想半年攒1万"         →  储蓄目标 Agent
"基金是什么"            →  理财教育 Agent
"这个能不能买"          →  消费决策（走分析 Agent）
```

**Supervisor 是额外的一层**，它不面向用户，纯做路由。这样用户只感知"一个 AI 助手"，但背后是 7 个专业 Agent。

风控 Agent 更特殊——它在所有 Agent 之后、用户看到之前。用户完全无感知。

---

### Q6：Supervisor 路由错了怎么办？有没有兜底机制？

**回答：**

有，两层兜底：

**第一层：规则层前置（最新优化）**
Supervisor 不是每次都调 LLM。我设计了一个 `quickClassify()` 函数，用多组关键词组合匹配：

- 要求组内所有关键词都命中才算匹配（降低误判）
- 置信度 ≥0.90 才跳过 LLM
- 大约 70% 的常见问题走规则，延迟 0ms，0 token 消耗

**第二层：降级兜底**
如果 LLM 返回的 JSON 解析失败，默认路由到 `general`，返回功能引导文案，不会报错或卡死。

**极端场景处理**：
- 用户说「帮我推荐一只股票」→ Supervisor 路由到 `finance_edu` → 理财 Agent 回答中不含推荐 → 风控 Agent 检查通过
- 如果 Supervisor 错误路由到 `record`（当成记账了）→ 用户看到的是「识别到以下支出...」这其实也很安全，不会有风险

对于财务场景，**错误路由到安全答案比错误路由到风险答案好得多**。这是产品设计的有意偏向。

---

### Q7：你是怎么设计每个 Agent 的 System Prompt 的？

**回答：**

我遵循一个固定模板，每个 Agent 的 Prompt 包含 5 个要素：

```
1. 角色定义    — "你是 MoneyMate 的 XX Agent"
2. 能力边界    — "你可以...你不可以..."
3. 数据注入    — 把用户真实财务数据拼进 Prompt
4. 输出格式    — JSON Schema 约束，保证前端可解析
5. 规则约束    — 该 Agent 特有的安全规则
```

**关键设计原则**：

**数据注入优先于 LLM 知识**。消费分析师不会直接"猜"用户花了多少钱，而是：

```
用户分类预算情况：
- 饮食：已花 ¥724 / 预算 ¥1800（40%）
- 购物：已花 ¥649 / 预算 ¥800（81%）
- ...

（这些都是从数据库查出来的，不是 LLM 编的）
```

**结构化输出约束**。所有 Agent 都强制 JSON 输出（`response_format: { type: 'json_object' }`），前端代码可以直接 `JSON.parse()` 而不用做自然语言解析。

**温度参数分层**：
- Supervisor：temperature=0.3（需要确定性路由）
- 分析类 Agent：temperature=0.5（需要一些表达灵活性）
- 风控 Agent：temperature=0.1（需要最确定性判断）

## 3. LLM 工程实践

### Q8：为什么选择 DeepSeek，而不是 GPT-4 或 Claude？

**回答：**

三个原因：

**1. 成本**
DeepSeek 的价格是 GPT-4 的 1/20 左右。对于 C 端产品，如果每个用户每天 5 次 AI 对话，成本必须在可控范围内。

**2. OpenAI 兼容**
DeepSeek 的 API 完全兼容 OpenAI SDK，我只需要改 `baseURL` 一行代码就能切换：

```typescript
const deepseek = new OpenAI({
  apiKey: process.env.DEEPSEEK_API_KEY,
  baseURL: 'https://api.deepseek.com',
});
```

这意味着未来切换到 Claude 或 GPT-4o 的成本接近于零。

**3. 中文能力**
DeepSeek 对中文财务术语的理解很好，"奶茶""外卖""打车"这种消费场景词汇识别准确。

**但有一个取舍**：DeepSeek 的 JSON 输出偶尔会有格式问题，所以我在 `tryParseJSON()` 里做了容错——先直接 parse，失败了再做正则提取 `{...}` 块。

---

### Q9：Agent 怎么处理 LLM 的幻觉问题？

**回答：**

这是做 AI 财务产品最核心的挑战——**财务数据不能编**。我的策略是三层防控：

**第一层：数据注入，不让 LLM 裸跑**
所有数字来自后端注入到 Prompt 的数据，不是 LLM 的知识。Agent 的 Prompt 里写着：

> 只能引用下方提供的真实财务数据，绝对不能编造数据

**第二层：代码与 Agent 的职责分离**
```
代码负责：金额计算、预算判断、图表数据、通知触发
Agent 负责：自然语言理解、分析解释、建议生成
```

这样即使 Agent 幻觉了，金额也是对的（因为金额是代码算的、Agent 只是引用了代码注入的数据）。

**第三层：风控 Agent 兜底**
风控 Agent 专门检查"是否编造了不存在的数据"。虽然目前规则还比较基础，但架构上预留了这个检查入口。

---

### Q10：流式输出是怎么实现的？为什么要做这个？

**回答：**

**实现方式**：

1. DeepSeek API 设置 `stream: true` → 返回 SSE 流
2. Vercel Serverless Function 将流逐块转发给前端（`Content-Type: text/event-stream`）
3. 前端用 `ReadableStream` + `getReader()` 逐块读取
4. 每收到一个 token，React 状态更新 → 用户看到逐字出现

**为什么要做**：

- 用户等待 3-5 秒看到空白的体验非常差，会怀疑"是不是坏了"
- 流式输出让用户看到 AI 在"思考"和"写"，心理等待感大幅降低
- 实测：同样的回答，流式输出的感知等待时间是 1 秒，非流式是 5 秒

**技术挑战**：Vercel Serverless Function 默认会缓冲响应。需要设置 `X-Accel-Buffering: no` 和 `Connection: keep-alive` 来强制实时推送。

## 4. 风控与合规

### Q11：一个财务类 AI 产品，最大的合规风险是什么？你是怎么应对的？

**回答：**

最大的风险是：**用户把 AI 当投资顾问，AI 给了不适当的建议导致用户亏钱**。

应对策略分为**预防**和**拦截**两层：

**预防层（Prompt 层）**：
理财 Agent 的 System Prompt 里写了 7 条红线+黄线规则：
- 红线：不推荐股票/基金代码、不承诺收益、不诱导借钱投资
- 黄线：必须含风险提示、必须先问应急备用金、必须问风险偏好

**拦截层（风控 Agent）**：
理财 Agent 的输出 → 风控 Agent 扫描 → 检查是否违规 → 有违规就修正后返回

例如用户问「推荐一只翻倍股票」：
- Supervisor 路由到理财 Agent
- 理财 Agent 给出标准教育回复（不推荐具体产品）
- 风控 Agent 再检查一遍，确认合规

**但坦率说，当前的风控还不够**。真正上线时还需要：
- 高风险关键词硬拦截（正则 + 名单，不走 LLM）
- 敏感回答人工抽检机制
- 用户投诉 → 回溯该用户所有对话的能力

---

### Q12：如果用户问「我应该买哪只股票」，你的系统会怎么处理？

**回答：**

这是一个典型的红线问题。处理链路：

```
用户：「我应该买哪只股票？」
  ↓
Supervisor → intent = finance_edu
  ↓
理财教育 Agent（Prompt 约束：不推荐具体股票）
  ↓
回答："我不能推荐具体股票...建议你先建立应急备用金,
       了解风险偏好,从低风险产品开始学习...⚠️风险提示..."
  ↓
风控 Agent 审查 → 无违规 → 返回
```

如果理财 Agent 罕见地输出了具体推荐（比如幻觉），风控 Agent 会检测到 `status: violation`，然后用修正后的回答替换。

**核心思路**：不是靠一个完美的 Prompt 防住所有问题，而是**两层防御**——理财 Agent 尽量不给，风控 Agent 确保没有。

## 5. 产品指标与评测

### Q13：你怎么衡量这个产品的 AI 做得好不好？

**回答：**

我把 AI 评测分成三个维度：

**1. 准确性（能不能用）**
| 指标 | 评测方式 |
|------|---------|
| 意图识别准确率 | 150-200 条样本，对比标注结果 |
| 记账提取准确率 | 金额/分类/时间三项的准确率 |
| 数据引用错误率 | 检查 AI 是否引用了不存在的财务数据 |

**2. 有用性（好不好用）**
| 指标 | 评测方式 |
|------|---------|
| 预算建议采纳率 | 用户是否按 AI 建议调整了预算 |
| AI 回复追问率 | 用户看完回答是否愿意继续聊 |
| 消费分析满意度 | 用户打分（👍/👎） |

**3. 安全性（敢不敢用）**
| 指标 | 评测方式 |
|------|---------|
| 高风险建议拦截率 | 用 20 条诱导性样本测试 |
| 收益承诺拦截率 | "稳赚""保本"类表述是否被拦截 |
| 风险提示完整率 | 理财回答是否都含风险提示 |

**评测集设计**：按 PRD 规划 150-200 条样本，覆盖 6 种意图 + 异常表达 + 高风险诱导。这在 V1 可以跑第一批数据。

### Q14：你怎么定义这个产品的北极星指标？

**回答：**

两个阶段，两个指标：

**MVP 阶段：连续记账天数**
- 记账类产品的核心瓶颈不是功能，是持续使用
- 如果用户能不中断地记 7 天，说明产品提供了持续价值
- 7 天留存率 >30% 就验证了 PMF

**V1 阶段：AI 助手使用率**
- AI 是我们的核心差异化，如果用户只用记账功能而不用 AI，产品就退化成传统记账工具
- 目标：周活跃用户中 >50% 用过 AI 助手
- AI 的使用频次和追问率是产品价值的直接证明

## 6. 技术决策与权衡

### Q15：为什么前端用 React + Context 而不是更重度的状态管理？

**回答：**

**场景决定架构**。这个项目是单用户前端、前端只做展示和交互，真正的状态在未来会迁移到后端数据库。

React Context + useReducer 对于当前场景完全够用：
- 状态结构清晰（用户、收入、支出、预算、提醒）
- 不需要中间件（不需要 saga/thunk 处理异步）
- Bundle 体积小（不引入 Redux/Zustand 额外依赖）

如果未来做多用户、多账本、实时同步，再切换到状态管理库——但 MVP 阶段过度设计是浪费。

---

### Q16：为什么部署选择 Vercel，而不是自己搭服务器？

**我的选择逻辑**：

| 需求 | Vercel | 自建服务器 |
|------|--------|-----------|
| 静态页面托管 | ✅ 原生支持 | 需要 Nginx 配置 |
| Serverless 函数 | ✅ 零配置 | 需要自己搭 |
| SSL/HTTPS | ✅ 自动 | 需要配置 |
| CI/CD | ✅ Git push 即部署 | 需要自己搭 |
| 成本 | 免费额度 100GB/月 | 至少 ¥50/月 |
| 冷启动 | 有（~500ms） | 无 |

对于 MVP，Vercel 是最优解。等用户量到一定规模，可以考虑迁移。

---

### Q17：当前架构最大的技术债是什么？

**回答：**

两个：

**1. 数据不持久化**
现在所有数据是前端 mock，刷新就没了。真正的产品必须接数据库（Supabase/Firebase），并且 Agent 读取的数据要从数据库实时查询，而不是硬编码的 mock。

**2. Agent Prompt 分散在两处**
`server/agents/` 有一套（Express 用），`api/chat.ts` 里又内联了一套（Vercel Serverless 用）。如果改一个 Prompt，要改两个地方。理想情况是抽一个共享的 Prompt 层。

这两个是上线前必须解决的技术债。

## 7. 用户体验设计

### Q18：首页只展示「今天还能花多少钱」，够吗？

**回答：**

这是有意为之的"一刀流"设计。

用户打开记账 App 的心理状态不是"我想看我的财务全景"，而是**"我现在还能花多少？"**。首页所有模块都在回答这个问题：

- 本月收入 → 池子里有多少
- 本月支出 → 已经花了多少
- 剩余预算 → 还能花多少
- 今日可花 → **今天还能花多少（核心答案）**
- 分类进度条 → 哪些分类快超了
- AI 建议 → 需要注意什么

给了用户一个每天打开的理由：看看今天还有多少额度。

---

### Q19：AI 记账为什么不直接自动写入，还要用户点确认？

**回答：**

这是一个故意的摩擦设计。

AI 识别不是 100% 准确的。如果自动写入：
- 分类错了 → 用户不知情 → 统计数据失真
- 金额错了 → 预算判断错误 → 引发连锁错误

所以设计成：
1. AI 提取 → 2. 用户确认 → 3. 写入

这个确认步骤的摩擦成本很低（一次点击），但避免了静默错误的成本很高（统计数据污染）。

**未来的优化方向**：如果用户对某个分类的 AI 判断连续 10 次都是对的，可以提示用户"以后自动记录不用确认了"。

## 8. 商业与增长

### Q20：这个产品怎么赚钱？

**回答：**

参考 PRD 的分层策略：

**免费版**：手动记账 + 基础预算 + 基础统计 + 每月 20 次 AI 问答
- 目的是让用户上手，养成习惯

**Pro 版（¥19/月 或 ¥149/年）**：无限 AI + 月度 AI 报告 + 自然语言记账 + 多账本
- AI 是核心付费点，因为边际成本不为零（每次调 LLM 都有成本）

**关键定价逻辑**：
- ¥19/月 = 每月省一顿外卖的钱
- 如果 AI 帮用户每月多省 ¥200，这个 ¥19 的 ROI 是 10 倍
- 定价锚定"帮用户省钱"而不是"AI 技术费"

## 9. 个人思考与复盘

### Q21：做完这个项目，你对 AI 产品最大的认知变化是什么？

**回答：**

最大的认知是：**AI 产品最难的不是接入 LLM，而是定义清楚 LLM 不该做什么**。

这个项目花了大量精力在：
- 哪些事让代码做、哪些事让 Agent 做
- Agent 的边界在哪里
- 幻觉怎么防
- 风控怎么加

接入 LLM 只需要 3 行代码。但让 LLM 在限定边界内安全、可靠地工作，才是真正的产品设计。

**第二个认知**：Prompt Engineering 不是写一段话那么简单。它是一个系统工程：
- System Prompt 设计（角色 + 边界 + 格式）
- 数据注入（把真实数据拼进去，不让 LLM 裸跑）
- 规则层辅助（先规则后 LLM，省成本降延迟）
- 输出后验证（风控检查、JSON 解析容错）

---

### Q22：如果重新做这个项目，有什么会做得不一样？

**回答：**

三件事：

**1. 先做评测集，再做 Agent**
现在 Agent 调好了但没有量化评测数据。应该先准备 150-200 条测试样本，每轮调完 Prompt 跑一遍评测，知道提升了多少、哪里还差。

**2. 后端数据先行**
现在 mock 数据做前端演示很好，但 Agent 读取 mock 数据而不是真实数据库，导致 Vercel 线上版 Agent 和本地开发版用的是同一套静态数据。应该先接一个轻量数据库。

**3. 流式输出一开始就该做**
非流式等了 5 秒，用户以为坏了。流式是 AI 产品的"必备体验"，不是"加分项"。

### Q23：你觉得这个产品最有价值的模块是哪个？

**回答：**

**不是 AI 助手，是首页看板上的「今日建议可花」**。

原因：
- 它给了用户每天打开 App 的理由
- 它背后是剩余预算 / 剩余天数的计算，简单但直观
- 它把复杂的财务数据翻译成了一句人话

AI 助手是锦上添花，但「今日可花」是雪中送炭。最好的 AI 产品不是让用户和 AI 聊天，而是让用户在不知不觉中受益于 AI。
